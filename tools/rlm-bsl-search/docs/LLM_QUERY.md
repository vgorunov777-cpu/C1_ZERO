# Настройка и механика llm_query

## Что это и зачем

В песочнице rlm-tools-bsl есть хелперы `llm_query(prompt, context)` и `llm_query_batched(prompts, context)` — они вызывают «маленькую» LLM прямо из `rlm_execute`, не возвращаясь в основной контекст агента. Это **вспомогательная** LLM, которая работает на стороне сервера.

### Когда это полезно

Агент нашёл 50 процедур в модуле и хочет классифицировать их на «типовые» и «нетиповые». Без `llm_query` он должен вернуть все 50 описаний в свой контекст и проанализировать сам — это ~5000 токенов на одну операцию. С `llm_query` классификация происходит на сервере, а в контекст агента попадает только финальный результат: «12 нетиповых процедур с префиксом ал_».

### Какие преимущества даёт

1. **Экономия контекста основной модели** — промежуточные данные (списки процедур, тексты методов, метаданные) не попадают в контекстное окно агента. В контекст возвращается только `print()` — компактный итог
2. **Ранжирование и фильтрация на сервере** — агент может попросить sub-LLM выбрать из 100 результатов grep 5 наиболее релевантных, отсеять комментарии и строковые литералы
3. **Батч-обработка** — `llm_query_batched` параллельно отправляет несколько промптов (до 8 потоков), что быстрее последовательных вызовов основной модели
4. **Дешёвые модели для рутины** — классификация, суммаризация, извлечение ключевых слов не требуют мощной модели. Можно использовать бесплатные или дешёвые модели (Ollama локально, free-тир OpenRouter)

### Пример: классификация процедур

```python
# Агент выполняет в rlm_execute:
procs = extract_procedures("Documents/РеализацияТоваровУслуг/Ext/ObjectModule.bsl")
names = [p['name'] for p in procs]

result = llm_query(
    "Раздели эти процедуры на типовые (из БСП/ERP) и нетиповые (кастомные доработки). "
    "Верни только нетиповые.",
    context="\n".join(names)
)
print(result)
# → "ал_РассчитатьСкидку, ал_ЗаполнитьДопРеквизиты, ал_ПроверитьЛимиты"
```

В контекст агента попало ~50 символов вместо ~3000.

## Настройка LLM-провайдера

Поддерживаются два варианта подключения (достаточно одного):

### OpenAI-совместимый endpoint (OpenRouter, LiteLLM, Ollama, vLLM)

```bash
# Windows
set RLM_LLM_BASE_URL=http://localhost:11434/v1
set RLM_LLM_API_KEY=
set RLM_LLM_MODEL=qwen2.5:7b

# Linux/macOS
export RLM_LLM_BASE_URL=http://localhost:11434/v1
export RLM_LLM_API_KEY=
export RLM_LLM_MODEL=qwen2.5:7b
```

| Переменная | Обязательность | Описание |
|---|---|---|
| `RLM_LLM_BASE_URL` | обязателен | Базовый URL endpoint'а |
| `RLM_LLM_MODEL` | обязателен | Имя модели |
| `RLM_LLM_API_KEY` | опционален | API-ключ (может быть пустым, например для Ollama) |

Требует пакет `openai` (входит в основные зависимости, ставится автоматически).

### Anthropic API

```bash
# Windows
set ANTHROPIC_API_KEY=sk-ant-api03-...

# Linux/macOS
export ANTHROPIC_API_KEY=sk-ant-api03-...
```

| Переменная | Обязательность | Описание |
|---|---|---|
| `ANTHROPIC_API_KEY` | обязателен | Ключ с [console.anthropic.com](https://console.anthropic.com) → API Keys |
| `RLM_SUB_MODEL` | опционален | Модель (по умолчанию Claude Haiku) |

### Приоритет провайдеров

Если заданы и `RLM_LLM_BASE_URL`, и `ANTHROPIC_API_KEY` — приоритет у OpenAI-совместимого endpoint'а.

## Как передать переменные окружения

- **stdio** — через секцию `env` в конфиге MCP (см. [INSTALL.md](INSTALL.md))
- **StreamableHTTP** — через файл `.env` рядом с рабочим каталогом, откуда запускается сервер. Сервер вызывает `load_dotenv(override=True)` при старте

Пример `.env`:
```
RLM_LLM_BASE_URL=https://api.kilo.ai/api/gateway
RLM_LLM_API_KEY=your-api-key
RLM_LLM_MODEL=minimax/minimax-m2.5:free
```

## Техническая механика

### llm_query(prompt, context)

1. Проверяет квоту LLM-вызовов сессии (зависит от effort-уровня)
2. Если передан `context`, формирует промпт: `"Context:\n{context}\n\nQuestion: {prompt}"`
3. Отправляет запрос в настроенный LLM-провайдер
4. Ограничение ответа: **max_tokens=1024** на каждый вызов
5. Возвращает текст ответа как строку

### llm_query_batched(prompts, context)

1. Принимает список промптов и общий контекст (применяется ко всем)
2. Параллельно отправляет запросы через `ThreadPoolExecutor(max_workers=8)`
3. Если один промпт упал с ошибкой — остальные продолжают работать, для упавшего возвращается `"[ERROR] ..."`
4. Результаты возвращаются в порядке исходных промптов

### Квоты по effort-уровням

| Effort | Макс. LLM-вызовов | Макс. rlm_execute |
|--------|-------------------|-------------------|
| low    | 5                 | 10                |
| medium | 15                | 25                |
| high   | 30                | 50                |
| max    | 50                | 100               |

`llm_query` резервирует 1 вызов, `llm_query_batched(prompts)` — `len(prompts)` вызовов. При превышении квоты — `RuntimeError`.

## Без настройки LLM

**Всё остальное работает нормально** — `find_module`, `grep`, `read_file`, `parse_object_xml` и все прочие хелперы не требуют API-ключа. Просто `llm_query()` будет недоступен.

Базовая функциональность rlm-tools-bsl не пострадает. Однако в отдельных сложных случаях основная модель-анализатор получит неранжированный ответ и потратит больше токенов на поиск сути — вместо того чтобы получить компактную выжимку от sub-LLM.
