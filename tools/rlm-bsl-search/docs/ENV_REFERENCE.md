# Справочник переменных окружения

Все переменные окружения, поддерживаемые rlm-tools-bsl. Ни одна из них не является обязательной — сервер запускается и работает с настройками по умолчанию.

Переменные можно задать:
- В файле `.env` рядом с рабочим каталогом (для StreamableHTTP — `load_dotenv(override=True)` при старте)
- В секции `env` конфига MCP (для stdio-транспорта)
- В окружении системы

Пример `.env` — см. файл `.env.example` в корне репозитория.

**Парсинг inline-комментариев в `.env`:**
- Значения в кавычках: `#` внутри сохраняется — `PASSWORD="p@ss#word"` → `p@ss#word`
- Без кавычек: inline-комментарий только через ` #` (с пробелом) — `URL=https://host#frag` → сохраняется, `KEY=value # comment` → `value`
- Экранированные кавычки внутри значений (`\"`) не поддерживаются — используйте одинарные кавычки или избегайте кавычек в значении

### Пример рабочей конфигурации

```env
# LLM через OpenAI-совместимый gateway (Kilo AI, free-модель)
RLM_LLM_BASE_URL=https://api.kilo.ai/api/gateway
RLM_LLM_API_KEY=your-kilo-api-key
RLM_LLM_MODEL=kilo-auto/free

# Индексы хранятся в отдельном каталоге (не в домашней папке)
RLM_INDEX_DIR=D:\Indexes\1C

# Проверка свежести индекса — расширенная выборка
RLM_INDEX_MAX_AGE_DAYS=7
RLM_INDEX_SAMPLE_SIZE=20
RLM_INDEX_SAMPLE_THRESHOLD=30

# Полное логирование хелперов (для отладки)
RLM_LOG_HELPERS=all
```

## LLM-провайдер для llm_query()

Хелперы `llm_query()` и `llm_query_batched()` вызывают вспомогательную LLM прямо из песочницы. Без настройки провайдера эти хелперы недоступны, всё остальное работает нормально. Подробнее — [LLM_QUERY.md](LLM_QUERY.md).

### OpenAI-совместимый endpoint (приоритетный)

| Переменная         | По умолчанию | Описание                                                                                                                                                                      |
| ------------------ | ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `RLM_LLM_BASE_URL` | —            | Базовый URL endpoint'а (например `http://localhost:11434/v1` для Ollama, `https://openrouter.ai/api/v1` для OpenRouter). Если задан — активирует OpenAI-совместимый провайдер |
| `RLM_LLM_MODEL`    | —            | Имя модели (обязателен при заданном `RLM_LLM_BASE_URL`). Примеры: `qwen2.5:7b`, `minimax/minimax-m2.5:free`                                                                   |
| `RLM_LLM_API_KEY`  | `""`         | API-ключ. Может быть пустым (Ollama, vLLM без авторизации)                                                                                                                    |

### Anthropic API (резервный)

| Переменная          | По умолчанию                | Описание                                                                                                                                       |
| ------------------- | --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| `ANTHROPIC_API_KEY` | —                           | API-ключ с [console.anthropic.com](https://console.anthropic.com) → API Keys. Активирует Anthropic-провайдер, если `RLM_LLM_BASE_URL` не задан |
| `RLM_SUB_MODEL`     | `claude-haiku-4-5-20251001` | Модель Anthropic для sub-LLM запросов                                                                                                          |

> Если заданы и `RLM_LLM_BASE_URL`, и `ANTHROPIC_API_KEY` — приоритет у OpenAI-совместимого endpoint'а.

## Сервер и сессии

| Переменная            | По умолчанию | Описание                                                                                                                               |
| --------------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| `RLM_MAX_SESSIONS`    | `5`          | Максимальное количество одновременных сессий                                                                                           |
| `RLM_SESSION_TIMEOUT` | —            | Единый таймаут сессии (минуты). Если задан — переопределяет оба таймаута ниже (backward compat)                                        |
| `RLM_SESSION_TIMEOUT_IDLE` | `10`    | Таймаут idle-сессии (минуты) — сессия без единого execute-вызова (calls=0)                                                              |
| `RLM_SESSION_TIMEOUT_ACTIVE` | `30`  | Таймаут активной сессии (минуты) — сессия с execute_calls>0                                                                             |
| `RLM_TRANSPORT`       | `stdio`      | Транспорт MCP-сервера: `stdio` или `streamable-http`. Можно также задать через аргумент `--transport` при запуске                      |
| `RLM_HOST`            | `127.0.0.1`  | Адрес для HTTP-транспорта. Можно также задать через `--host`                                                                           |
| `RLM_PORT`            | `9000`       | Порт для HTTP-транспорта. Можно также задать через `--port`                                                                            |
| `RLM_CONFIG_FILE`     | —            | Путь к `service.json`. Каталог этого файла определяет расположение `projects.json`, `.env`, `logs/` и `cache/`. Если не задан — `~/.config/rlm-tools-bsl/service.json` для конфига и `~/.cache/rlm-tools-bsl/` для кэша |

## Docker

Переменные для работы в Docker-контейнере.

| Переменная     | По умолчанию | Описание                                                                                                                                                                   |
| -------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `RLM_VERSION`  | —            | Версия пакета для установки (в `docker-entrypoint.sh`). Пусто или `latest` — авто-обновление из PyPI при каждом старте. Конкретная версия (напр. `1.6.2`) — `pip install ==version`, без проверки PyPI. Позволяет откатиться при сломанном релизе |
| `RLM_PATH_MAP` | —            | **Трансляция хостовых путей в контейнерные.** Формат: `хостовый_префикс:контейнерный_префикс`. Пример: `D:/Repos/1c:/repos`. Когда задана, пользователь может указывать хостовые пути при регистрации проектов, старте сессий и работе с индексами — сервер автоматически подменяет префикс. Без этой переменной нужно использовать контейнерные пути (`/repos/...`) |

## SQLite-индекс методов

Индекс опционален — создаётся командой `rlm-bsl-index index build <path>`. Подробнее — [INDEXING.md](INDEXING.md).

| Переменная                    | По умолчанию              | Описание                                                                                                                                    |
| ----------------------------- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `RLM_INDEX_DIR`               | см. precedence ниже       | Каталог хранения индексов. **Precedence (v1.9.2+):** ① если задан `RLM_INDEX_DIR` — используется он; ② иначе если задан `RLM_CONFIG_FILE` — `dirname(RLM_CONFIG_FILE)/index/` (для Windows-службы LocalSystem пишет рядом с config/logs/cache, а не в `system32`); ③ иначе — `~/.cache/rlm-tools-bsl/`. Внутри создаётся подпапка с хешем пути конфигурации, например: `<root>/a3f8b2c1/bsl_index.db`. При первом старте v1.9.2 legacy-индексы из `~/.cache/rlm-tools-bsl/<hash>/` автоматически переезжают в новый root |
| `RLM_INDEX_MAX_AGE_DAYS`      | `7`                       | Порог предупреждения о возрасте индекса (дни). Если индекс старше — статус `STALE_AGE`                                                      |
| `RLM_INDEX_SAMPLE_SIZE`       | `5`                       | Количество файлов для выборочной проверки свежести. `0` — отключить проверку                                                                |
| `RLM_INDEX_SAMPLE_THRESHOLD`  | `30`                      | Минимальное число модулей в индексе, при котором выполняется выборочная проверка                                                            |
| `RLM_INDEX_SKIP_SAMPLE_HOURS` | `24`                      | Если индекс моложе этого порога (часы), выборочная проверка пропускается — сразу `FRESH`                                                    |
| `RLM_CACHE_MAX_AGE_DAYS`      | `14`                      | Порог авто-очистки **кэша** BSL-файлов (`~/.cache/rlm-tools-bsl/` либо `dirname(RLM_CONFIG_FILE)/cache/`, если задан). При старте MCP-сервера удаляются подпапки проектов, не использовавшихся дольше этого срока. Значение `0` или отрицательное — очистка отключена. Активность проекта обновляется при каждом `rlm_start`, `rlm_index build/update/info`. **Миграция с v1.9.0**: при первом запуске v1.9.1 legacy-кэши зарегистрированных проектов (без маркера `last_used.txt`) получают свежий маркер и не удаляются. Зарегистрированные-но-забытые проекты чистятся на последующих запусках, когда маркер устареет. **BSL-индексы (`RLM_INDEX_DIR`) НЕ подпадают под эту очистку** — они дорого строятся; управляйте ими вручную через `rlm_index(action='drop')` |

## Реестр проектов

Файл `projects.json` хранит соответствие человекочитаемых имен проектов путям к каталогам исходников. Располагается **рядом с активным `service.json`**:

- Если задан `RLM_CONFIG_FILE=/etc/rlm-tools-bsl/service.json` -- реестр будет `/etc/rlm-tools-bsl/projects.json`
- Иначе -- `~/.config/rlm-tools-bsl/projects.json`

Управление реестром -- через MCP-тул `rlm_projects` (list/add/remove/rename/update). После регистрации проекта можно открывать сессии по имени: `rlm_start(project='My Config', query='...')`.

## Диагностика

| Переменная        | По умолчанию | Описание                                                                                                                              |
| ----------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| `RLM_LOG_HELPERS` | —            | Режим логирования хелперов. По умолчанию логируются только вызовы дольше 0.1 сек. Значение `all` — логировать все вызовы с таймингами |
